# 轻小说对话说话人标注方案

基于以下材料整理：

- `prompt.txt`
- `memory-skills-summary.md`
- `test_ollama.py`

本文不是代码实现，而是一个可直接落地的设计方案。目标是先把路线定清楚，后面再按阶段实现。

## 1. 目标与边界

### 目标

把轻小说中的对话自动加上说话人标签，例如：

```text
【罗伦斯】「要不要来一颗？」
【骑士】「呃……」
```

### 非目标

- 这一版不追求 100% 全自动无人工复核。
- 这一版不建议直接原地改原文。
- 这一版不让大模型直接输出整段加标签后的全文，而是让模型输出结构化判断，再由 Python 写回文本。

## 2. 先给结论

### 我建议的总体方案

不要在“单阶段直接标注”和“两阶段先扫描再标注”之间二选一，而是采用一个更稳的 **三阶段混合方案**：

1. **预处理阶段**
   把小说拆成章节、场景、段落、对话单元，建立稳定 ID。
2. **发现阶段**
   扫描全卷，抽取角色、别名、称谓、场景参与者、未知角色临时 ID、说话风格线索，先建记忆，不急着输出最终标签。
3. **标注阶段**
   对每句对话，用“局部上下文 + 检索出的记忆”交给多个模型判断，聚合结果。
4. **回填修正阶段**
   如果后文揭示了先前“未知角色”的真实身份，再回填前文，把临时 ID 合并成正式角色名。

这个方案比纯单阶段更稳，也比严格两阶段更灵活。

## 3. 为什么不能简单地“一句一句丢给模型”

你的直觉是对的。问题不只是上下文太短，而是轻小说里的说话人判断本质上依赖四类信息：

1. **局部动作线索**
   例如“罗伦斯把糖递给骑士”后接“要不要来一颗？”，这句话大概率就是罗伦斯说的。
2. **场景参与者**
   当前场景有哪些人在场，谁刚刚开口，谁正在回应谁。
3. **长期角色知识**
   例如角色别名、口癖、身份、与谁同行。
4. **延后揭示**
   有些人第一次说话时就是神秘人物，名字要很后面才揭示。

所以不能把任务理解成“分类一句话是谁说的”，而应该理解成：

**在一个持续演化的叙事状态里，对每个对话单元做身份推断。**

## 4. 推荐的系统架构

### 4.1 主架构原则

我不建议让某个大模型做“总负责人 + 工具调用总控”。  
更稳的做法是：

- **Python 是总控**
- **本地大模型只做受约束的子任务**

原因很简单：

- 工具编排、文件写入、重试、缓存、投票、回填，本来就是 Python 更擅长。
- LLM 适合做“不完全信息下的语义判断”，不适合负责整个工程控制流。
- 这样更容易调试，也更省算力。

### 4.2 角色分工

建议这样分：

- **Python orchestrator**
  负责任务编排、切分、检索、调用模型、聚合投票、写文件。
- **worker models**
  负责“这句对话更可能是谁说的”。
- **memory summarizer / updater**
  负责把新发现的角色信息压缩进记忆卡片。
- **referee model**
  只在多模型分歧大、或者需要合并未知角色时介入。

也就是说，真正的“主负责人”应该是 Python 管线，不是某一个 LLM。

## 5. 记忆系统怎么设计

你提到的三个 memory skill 给出的启发非常有价值，我建议吸收它们的优点，但不要照搬。

### 5.1 借鉴结论

- 从 `memory-setup` 借鉴：
  **先检索，再决策**，而不是把所有记忆一次性塞进 prompt。
- 从 `memory` 借鉴：
  **重要信息要结构化沉淀，并且立即写入索引。**
- 从 `memory-manager` 借鉴：
  **记忆必须分层，必须考虑压缩风险。**

### 5.2 我建议的四层记忆

#### A. 工作记忆 Working Memory

只服务当前一句对话，生命周期最短。

包含：

- 当前对话文本
- 当前段落原文
- 前后若干段叙述
- 前 2 到 4 句对话
- 当前场景摘要

#### B. 情节记忆 Episodic Memory

记录“刚刚发生了什么”，按场景或片段组织。

包含：

- 场景 ID
- 场景参与者
- 最近谁对谁说话
- 最近动作链
- 当前未解决的“未知角色”列表
- 场景摘要

这一层很像 `memory-manager` 里的 episodic。

#### C. 角色语义记忆 Semantic Memory

记录“这个角色是什么样的人”。

每个角色一张卡片，建议包含：

- `entity_id`
- `display_name`
- `aliases`
- `titles`
- `role`
- `gender_or_unknown`
- `speech_style`
- `relationship_hints`
- `first_appearance`
- `latest_appearance`
- `evidence_examples`
- `certainty`

对未知角色也建卡，但先用临时 ID，例如：

- `mystery_001`
- `mystery_merchant_002`

这一层是全卷最重要的长期记忆。

#### D. 规则记忆 Procedural Memory

记录“应该怎样标注”，不记录剧情事实。

包含：

- 输出 JSON 规范
- 什么情况下允许输出 `unknown`
- 什么情况下新建 `mystery_xxx`
- 什么情况下标为 `crowd` / `npc`
- 什么情况下进入人工复核队列
- 投票聚合规则

这一层很像 `memory-manager` 里的 procedural。

### 5.3 为什么这样分层

因为说话人标注不是单纯的知识问答，而是：

- 一部分依赖刚刚发生的事
- 一部分依赖角色长期特征
- 一部分依赖固定操作规则

这三种东西混在一起，会很快把 prompt 撑爆，也很难检索。分层以后，每次只取需要的那一部分。

## 6. 对“未知角色”的处理，我建议第三种方案

你提出了两个方案：

1. 直接允许模型输出“未知”
2. 先扫描整卷收集角色，再二次标注

我建议升级成第三种：

### **允许未知，但未知不是一个值，而是一类可追踪实体**

也就是说，不要只输出：

```text
未知
```

而要输出：

```text
mystery_003
```

或者最终写回文本时显示成：

```text
【未知人物#003】「……」
```

这样做有几个关键好处：

1. 它允许“先不知道名字，但知道是同一个人”。
2. 后文身份揭示后，可以把 `mystery_003 -> 赫萝` 统一回填。
3. 这样不会把所有未知人物混成一个“未知”。

所以我的建议是：

- **允许 unknown**
- **更进一步，允许 stable unknown identity**
- **再加一个后处理的身份合并阶段**

这是整套方案里最关键的设计之一。

## 7. 数据处理流程

## 7.1 阶段 0：预处理

输入是一整卷原文，输出是可追踪的结构化单元。

建议步骤：

1. 统一换行、标点、全半角。
2. 按章节切分。
3. 按段落切分。
4. 抽取所有 `「...」` 对话片段。
5. 给每个对话分配稳定 ID。
6. 记录对话所在段落、前后段落、章节、场景。

对话单元建议长这样：

```json
{
  "dialogue_id": "v01-c03-s02-d0017",
  "chapter_id": "c03",
  "scene_id": "s02",
  "paragraph_id": "p104",
  "text": "要不要来一颗？",
  "raw_paragraph": "罗伦斯拿出一颗放入口中，再把整袋蜂蜜糖递给骑士。「要不要来一颗？」",
  "prev_paragraphs": ["..."],
  "next_paragraphs": ["..."]
}
```

注意：  
**最小处理单位不是“句子”，而是“对话单元 + 邻近叙述”。**

## 7.2 阶段 1：发现与建库

这一阶段的目标不是给最终标签，而是先把记忆建起来。

主要任务：

1. 识别角色名、称谓、别名。
2. 识别场景参与者。
3. 抽取说话风格线索。
4. 识别新出现但未命名的角色，创建 `mystery_xxx`。
5. 记录人物关系和事件线索。
6. 生成场景摘要。

输出：

- `characters.jsonl`
- `aliases.jsonl`
- `scene_state.jsonl`
- `mystery_entities.jsonl`
- `semantic memory cards`

这一阶段可以用较大的上下文窗口，因为目标是“建卷级记忆”。

## 7.3 阶段 2：正式标注

对每个对话单元，Python 先检索最相关的信息，再让多个模型判断。

每次调用模型时，给它的上下文应该是：

1. 当前对话文本
2. 当前段落原文
3. 前后若干段叙述
4. 场景摘要
5. 场景当前活跃角色
6. Top-K 相关角色卡
7. Top-K 相关未知角色卡
8. 固定的输出规范

而不是把整卷记忆全塞进去。

## 7.4 阶段 3：回填修正

这一阶段专门解决：

- 前文是未知，后文才知道身份
- 早期误判，后期证据更充分
- 多个 `mystery_xxx` 其实是同一人

建议做法：

1. 先跑“未知角色合并”
2. 再跑“正式角色映射”
3. 最后重写最终标签文件

## 8. 多模型投票应该怎么做

你的方向是对的，但不要只做“多数票”。

### 8.1 推荐方案

对每句对话：

1. 调用 3 到 5 个模型独立判断
2. 每个模型输出结构化 JSON
3. Python 做加权聚合
4. 如果结论稳定，直接接受
5. 如果分歧大，送给 referee model
6. 如果 referee 也不稳，打入人工复核

### 8.2 为什么不用简单多数票

因为不同模型强弱不同，不能默认一票等权。

建议用一个小型人工标注验证集，先测每个模型在“说话人识别”任务上的表现，再决定权重。

例如：

- `qwen3:32b`：权重 1.0
- `llama3:latest`：权重 0.8
- `glm-4.7-flash:latest`：权重 0.7

只是示意，具体要实测。

### 8.3 如果本地模型数量不够

可以用“同模型多次运行”做备选，但我不建议单纯靠高 temperature。

更稳的替代方式是：

1. 同一模型，使用不同 prompt 版本
2. 同一模型，使用不同上下文窗口
3. 同一模型，低温重复采样

相比之下，直接把 temperature 拉高，容易让输出更散，但不一定更可靠。

### 8.4 一个可执行的判定规则

可以先用这种规则起步：

- Top1 加权得分 >= 0.65
- Top1 与 Top2 差值 >= 0.20
- 至少 2 个模型给出同一实体

满足则接受，否则进入仲裁或复核。

## 9. 输出格式怎么设计

这一点非常重要。  
我强烈建议 **只允许模型输出严格 JSON**，不要输出自然语言解释，更不要输出修改后的原文。

### 9.1 推荐字段

```json
{
  "dialogue_id": "v01-c03-s02-d0017",
  "speaker_entity_id": "char_lawrence",
  "speaker_display": "罗伦斯",
  "speaker_status": "known",
  "confidence": 0.87,
  "candidate_speakers": [
    {"entity_id": "char_lawrence", "score": 0.87},
    {"entity_id": "char_knight", "score": 0.11}
  ],
  "evidence": [
    "上一句叙述显示罗伦斯把糖递给骑士",
    "该问句更像递糖的一方发出"
  ],
  "should_create_new_entity": false,
  "new_entity_hint": "",
  "needs_review": false
}
```

### 9.2 `speaker_status` 建议取值

- `known`
- `mystery`
- `npc`
- `ambiguous`
- `review`

### 9.3 关键原则

1. **实体 ID 和显示名称分离**
   便于后续合并、改名、回填。
2. **必须保留候选项和置信度**
   方便后处理。
3. **必须允许 abstain**
   也就是允许“不确定，不瞎猜”。

## 10. 文件产物应该怎么存

### 10.1 不建议原地改原文

V1 最稳的做法：

- 原文不动
- 输出新文件
- 同时保存中间结构化结果

### 10.2 建议目录结构

```text
outputs/
  volume_01/
    source_copy.txt
    dialogues.jsonl
    votes.jsonl
    review_queue.jsonl
    final_labeled.txt
    memory/
      episodic/
      semantic/
      procedural/
```

### 10.3 为什么要保留中间文件

因为后面一定会遇到这些需求：

- 想重新跑某一章，不重跑全卷
- 想调 prompt，对比旧结果
- 想看哪几句最不确定
- 想回填某个未知角色

如果不保留中间层，调试会很痛苦。

## 11. 记忆压缩怎么做才不会爆

这是你担心的重点之一，我同意确实复杂，但可以控制。

### 11.1 核心原则

不是把记忆“压到最短”，而是把记忆压成 **可检索的结构**。

### 11.2 我建议的控制方式

#### 对 episodic memory

- 每个场景 1 份摘要
- 每个场景只保留最近若干轮对话状态
- 场景结束后冻结

#### 对 semantic memory

- 每个角色卡限制字段数量
- `evidence_examples` 只保留最有代表性的 3 到 5 条
- 别名与称谓单独索引

#### 对 procedural memory

- 固定模板，不频繁变动

### 11.3 检索时的预算控制

每次推理，不要把全部角色卡带进去，只带：

- 当前场景活跃角色
- 名字在邻近段落中出现过的角色
- 最近出现过的未知角色
- 检索分数最高的 Top-K 角色卡

这就是 `memory-setup` 给你的最大启发之一：  
**不是“记住全部”，而是“需要时能找回相关部分”。**

## 12. 对你提出的 6 个问题的直接回答

### 问题 1：一整卷太大，怎么办？

结论：  
不要按“整卷”输入，也不要机械地“一句一句孤立判断”。

建议：

- 先按章节和场景切
- 以“对话单元 + 邻近叙述”为最小推理单位
- 卷级信息通过记忆检索回补

### 问题 2：记忆怎么设计？

结论：  
用 **工作记忆 + 情节记忆 + 语义记忆 + 规则记忆** 四层结构。

重点：

- 不是做通用聊天记忆
- 是做“面向说话人识别”的任务型记忆
- 记忆一定要结构化，不能只是不断堆自然语言摘要

### 问题 3：多个本地模型投票是否可行？

结论：  
可行，而且值得做，但建议做 **加权投票 + 仲裁**，不要只做多数票。

补充：

- 如果模型不够多，可以同模型多次跑
- 但更建议“多 prompt / 多窗口”而不是单纯高温采样

### 问题 4：输出应该是“名字”还是“未知”？

结论：  
都要，但还不够。最好的做法是：

- 已知人物：输出正式角色实体
- 未知但可追踪：输出 `mystery_xxx`
- 完全无法判断：输出 `ambiguous` 或进入 `review`

所以不是“名字 vs 未知”二选一，而是三层：

- `known`
- `mystery`
- `review/ambiguous`

### 问题 5：是否需要一个总负责人模型？

结论：  
V1 不需要。

更推荐：

- Python 做总控
- 一个较强模型做 referee
- 其他模型做 worker

等系统稳定后，如果你确实想加“manager model”，也应该只让它做策略建议，不要让它接管全部控制流。

### 问题 6：Python + Ollama 是否合适？

结论：  
非常合适。

原因：

- Python 非常适合文本切分、状态管理、JSON 解析、文件写回
- 你已经有 Ollama 和模型样例
- 本地调用容易做缓存、重试、并行、超时控制

但要注意：

- 可用模型很多，不代表都适合文学判断
- `coder` 模型未必比通用模型更擅长说话人识别
- 最终要靠小规模验证集实测

## 13. 我最推荐的 V1 落地方案

如果目标是先做出一个“能跑、能评估、可迭代”的版本，我建议按这个顺序来。

### 第一步：先做一个小验证集

人工挑选：

- 1 到 3 个章节
- 手工标注说话人
- 同时标出“未知但可追踪”和“无法判断”

这一步非常关键，因为后面模型权重、prompt 优化、记忆设计都要靠它评估。

### 第二步：只做单模型 + 严格 JSON

先不要一上来就多模型投票。  
先验证这几件事：

- 对话抽取是否稳定
- 上下文窗口是否合适
- 输出 JSON 是否稳定
- `mystery_xxx` 机制是否有用

### 第三步：加入分层记忆

先加：

- scene summary
- character cards
- unknown entity cards

暂时不要做复杂的全局压缩。

### 第四步：再加多模型投票

等单模型链路跑通后，再加入：

- 多 worker
- 加权聚合
- referee
- review queue

### 第五步：最后做回填修正

这是提高质量的关键阶段，尤其适合轻小说这种“后文揭示身份”的叙事。

## 14. 我对整套方案的最终判断

这个项目是能做的，而且很适合本地 LLM + Python 的组合。

但它不是一个“直接喂小说 -> 模型吐全文”的项目，而更像一个：

- 带记忆检索的文本处理管线
- 带不确定性管理的实体识别系统
- 带回填修正的叙事状态机

如果路线选对，难点不在“能不能调用模型”，而在：

1. 对话单元怎么切
2. 记忆怎么分层
3. 未知角色怎么追踪
4. 不确定结果怎么处置
5. 后文揭示后怎么回填

我的建议非常明确：

- **不要原地改原文**
- **不要让 LLM 直接输出整本标注文本**
- **不要把“未知”设计成一个单值**
- **不要让单个大模型承担总控**
- **先做小样本评估，再扩大到整卷**

## 15. 一句话版方案

用 Python 作为总控，把小说拆成“对话单元 + 场景”，先跑一遍建角色记忆与未知角色临时 ID，再在“局部上下文 + 检索记忆”下让多个本地模型做结构化说话人判断，最后对未知角色做回填合并，并输出新文件而不是改原文。
